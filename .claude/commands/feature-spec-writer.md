# Feature Spec Writer

ざっくりした要望から実装仕様を起こす。

## Steps

1. ユーザーの要望を読む
2. **UI実装が含まれる場合は、実装前に以下を確認する**
   - 参考にしたいデザイン・アプリはあるか
   - 各画面で「見せるべき情報」と「見せなくていい情報」の区別
   - 選択肢が有限な入力項目はテキスト入力ではなくリスト/ピッカーにする
3. 以下のテンプレートで仕様書を作成する
4. MVP範囲とスコープ外を明確に分ける
5. 受け入れ条件をテスト可能な形で書く

## UI実装の禁止事項（過去の失敗より）
- 参考デザインを聞かずに実装しない
- ホーム/メイン画面に詳細情報を詰め込まない（最小限の情報だけ）
- 有限選択肢をテキスト入力にしない（路線名・カテゴリ等）
- 既存コードを「動いているから」という理由だけで残さない

## Output Template

```markdown
## 機能名
[機能の名前]

## 背景・目的
[なぜこの機能が必要か]

## ユーザーストーリー
[ユーザーとして、〇〇したい。なぜなら〇〇だから。]

## MVP範囲（今回実装する）
- [機能1]
- [機能2]

## スコープ外（今回実装しない）
- [機能X] — 理由

## 受け入れ条件
- [ ] [操作] したとき、[結果] になる
- [ ] [エラーケース] のとき、[エラーメッセージ] が表示される

## 関連ファイル（推定）
- lib/screens/xxx.dart
- lib/providers/xxx.dart

## 優先度
[High/Medium/Low] — [根拠]
```
