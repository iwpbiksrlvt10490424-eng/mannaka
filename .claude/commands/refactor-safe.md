# Refactor Safe

仕様を変えずにコードを改善する。テストが全パスのまま進める。

## Goal
動作を変えずにコードの可読性・重複・パフォーマンスを改善する。

## Steps

1. 改善対象を明確に1つに絞る（「命名改善のみ」「重複削除のみ」等）
2. 影響範囲を確認する
3. テストが全パスであることを確認してから変更を始める
4. 改善を実施する
5. `flutter test` 全パス / `flutter analyze` 0 issues を確認する
6. 変更内容を報告する

## What to Look For

- 同じロジックが複数箇所に書かれていないか（DRY違反）
- 意味が伝わらない変数名・関数名がないか
- build()内での重い計算がないか（final変数に切り出す）
- 不要なrebuildを引き起こすref.watchがないか
- `Future.delayed` の不必要な人工遅延がないか
- awaitで画面遷移をブロックしていないか（先に遷移→スケルトン表示が原則）

## Rules
- 仕様・動作を変えない
- 公開APIのシグネチャを変えない
- テストが通らなくなる変更はしない
- 指示なしで依存関係を追加しない
- 改善範囲を広げすぎない

## Output Format

```markdown
## 改善内容

### [改善1のタイトル]
- 対象: lib/xxx.dart
- 変更前: [概要]
- 変更後: [概要]
- 理由: [なぜ改善になるか]

## flutter test: [全パス / X件失敗]
## flutter analyze: [0 issues / X issues]

## 未実施の改善候補
- [候補] — [理由・推奨タイミング]
```
